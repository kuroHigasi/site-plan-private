# エラー設計テンプレート

> API全体で共通利用するエラーレスポンス、例外ハンドリング方針を整理するためのテンプレートです。
>
> 共通仕様: `01_DOCS/wiki/04_API設計/00_共通仕様.md`

---

## 1. 目的

- [ ] API全体で RFC7807 互換のエラー表現を統一する
- [ ] クライアントが `status` と `detail` で判定できる形式にする
- [ ] 実装・運用で迷わない応答形式にする

---

## 2. エラーレスポンス共通形式（RFC7807 互換）

```json
{
  "type": "about:blank",
  "title": "Bad Request",
  "status": 400,
  "detail": "{{ DETAIL_MESSAGE }}",
  "instance": "/api/{{ API_TYPE }}/{{ PATH }}.php{{ QUERY }}",
  "traceId": "{{ TRACE_ID }}"
}
```

### 記載ルール
- [ ] `type` は問題の種類（省略時 `about:blank` 可）
- [ ] `title` は HTTP ステータスの短い説明
- [ ] `status` は HTTP ステータスコード（body 内にも含める）
- [ ] `detail` は利用者・開発者向けの具体的メッセージ
- [ ] `instance` はリクエスト URI（デバッグ用）
- [ ] `traceId` はサーバー側ログ追跡用 ID
- [ ] DB 例外の詳細（SQL 文、スタックトレース）は **返さない**

---

## 3. エラー分類

| 分類 | 代表ステータス | 説明 |
|------|----------------|------|
| バリデーションエラー | 400 | 入力値不正、必須項目不足、JSON 不正 |
| 認証エラー | 401 | Cookie なし、ログイン失敗 |
| CSRF エラー | 403 | `X-CSRF-Token` 欠落/不一致 |
| 未検出エラー | 404 | リソース未存在 |
| メソッド不許可 | 405 | 許可外 HTTP メソッド |
| Accept 不正 | 406 | Accept ヘッダー未指定・形式不正・未対応バージョン |
| システムエラー | 500 | 予期しない内部エラー（詳細非公開） |

---

## 4. ステータスコード別定義

| ステータス | title 例 | detail 例 | クライアント対応 |
|-----------|---------|-----------|------------------|
| 400 | Bad Request | `page must be an integer of 1 or greater.` | 入力値を修正 |
| 401 | Unauthorized | `Authentication required.` | 再ログイン |
| 403 | Forbidden | `CSRF token mismatch.` | session.php で CSRF 再取得 |
| 404 | Not Found | `Changelog not found.` | リソース不存在を表示 |
| 405 | Method Not Allowed | `Method not allowed.` | メソッドを確認 |
| 406 | Not Acceptable | `Unsupported API version.` | Accept ヘッダーを確認 |
| 500 | Internal Server Error | `An unexpected error occurred.` | 汎用エラー表示、再試行 |

## 5. バリデーションエラー詳細

| フィールド | ルール | detail 例 |
|----------|--------|-----------|
| {{ FIELD_1 }} | {{ RULE_1 }} | `{{ FIELD_1 }} is required.` |
| {{ FIELD_2 }} | {{ RULE_2 }} | `{{ FIELD_2 }} must be between 1 and 50.` |

### バリデーションルール
- [ ] 必須チェック
- [ ] 型チェック
- [ ] 桁数チェック（例: title 最大 255 文字）
- [ ] 日付形式チェック（`YYYY-MM-DD` のみ）
- [ ] 相関チェック（例: `from <= to`）
- [ ] JSON body 空・不正チェック

---

## 6. 例外ハンドリング方針

- [ ] 業務例外とシステム例外を分ける
- [ ] ログ出力レベルを定義する
- [ ] クライアントへ返す情報の範囲を定義する（500 では詳細非公開）
- [ ] `traceId` とサーバーログを紐付ける
- [ ] 障害通知条件を定義する

---

## 7. ログ設計

| 項目 | 内容 |
|------|------|
| traceId | エラーレスポンスとサーバーログの紐付け |
| 出力タイミング | 4xx/5xx 発生時 |
| 出力項目 | traceId, status, instance, 内部エラー詳細 |
| 個人情報の扱い | ログに PII を平文で出力しない |
| 保持期間 | {{ RETENTION }} |

---

## 8. レビュー観点

- [ ] すべてのエラーが RFC7807 形式である
- [ ] ステータスコードと意味が一致している
- [ ] `detail` がフロント実装で使える粒度である
- [ ] 500 時に内部情報が漏れない
- [ ] 406（Accept 不正）の扱いが定義されている
